## Purpose

This document defines how GridPoint Energy, Inc. ("GridPoint") satisfies **CIP-007-6 Requirement R5 — System Access Control** for its **14 Medium-impact BES Cyber Systems (BCS)** and associated **EACMS**, **PACS**, and **PCA**. R5 governs authentication of user access, the elimination or control of default and shared/generic accounts, and password/authenticator parameters. Implementing R5 closes **GAP-16** (shared and default accounts present on control-center and substation BCS) from the Phase-02 gap register. R5 works in concert with CIP-004 access authorization (Phase 03) — CIP-004 decides *who* may have access; CIP-007 R5 enforces *how* that access is authenticated and controlled on the Cyber Asset.

## CIP-007-6 R5 Requirement Parts in Scope

| Part | Requirement (summary) | GridPoint Implementation | Evidence |
|---|---|---|---|
| **R5.1** | Have a method to enforce authentication of interactive user access, per Cyber Asset capability | All interactive access requires unique-account authentication; local consoles and the Intermediate System enforce credential prompts | Auth configuration; access-method register |
| **R5.2** | Identify and inventory all known enabled default or generic account types | Account inventory maintained per BCS/EACMS/PACS; default/generic accounts flagged and dispositioned | Account inventory register |
| **R5.3** | Identify individuals with authorized access to shared accounts | Where a shared account is unavoidable (e.g., an operator console requiring continuous availability), the named individuals authorized to it are recorded | Shared-account authorization list |
| **R5.4** | Change known default passwords, per Cyber Asset capability | Vendor/default passwords changed at commissioning; residual TFE-tracked exceptions documented where a device cannot change them | Default-password change log |
| **R5.5** | For password-only authentication, technically enforce password parameters: minimum length and minimum complexity (or per device capability) | Enforced minimum length of **8 characters** and complexity (three of four character classes) where technically feasible | Password-policy configuration |
| **R5.6** | Where technically feasible, enforce password changes at least once every **15 calendar months** | Directory-integrated accounts rotate on a ≤ 15-month cycle; device-local exceptions documented | Rotation policy; rotation evidence |
| **R5.7** | Where technically feasible, either limit unsuccessful authentication attempts or generate alerts after a threshold of unsuccessful attempts | Account lockout after 5 failed attempts on capable systems; SIEM alert (see CIP-007 R4) where lockout is not feasible | Lockout config; SIEM alert rule |

## No Shared, No Default Accounts — Control Statement

GridPoint's target state is **unique, individually attributable authentication** for every interactive user on every applicable Cyber Asset. The program applies the following hierarchy:

1. **Eliminate** default and generic accounts wherever the device permits (removes the R5.3/R5.4 exposure entirely).
2. **Change** the default password where the account cannot be removed (R5.4) and rotate it on the R5.6 cycle.
3. **Attribute** any residual shared account to a named, CIP-004-authorized individual list (R5.3), reviewed each calendar quarter alongside the CIP-004 R4 access-privilege verification.

This hierarchy is what closes **GAP-16**: legacy substation relays that shipped with vendor generic accounts have had those accounts disabled where possible, default passwords changed where not, and the small residual set is now inventoried and attributed.

## Password & Authenticator Parameters (R5.5)

| Parameter | GridPoint Standard | CIP-007-6 Basis |
|---|---|---|
| Minimum length | 8 characters (or maximum the device supports) | R5.5.1 |
| Complexity | 3 of 4 classes: uppercase, lowercase, digit, special (or maximum supported) | R5.5.2 |
| Maximum age | ≤ 15 calendar months (where feasible) | R5.6 |
| Failed-attempt handling | Lockout after 5 attempts, or SIEM alert | R5.7 |

Where a Cyber Asset is technically incapable of enforcing a parameter, the limitation is documented per the device-capability register, and — if the shortfall is against a strict requirement — a **Technical Feasibility Exception (TFE)** is filed with ReliabilityFirst. Interactive Remote Access additionally layers **multi-factor authentication** at the Intermediate System (CIP-005 R2, 04.03), which strengthens R5.1 for all remote sessions.

## Interactive Access Model

Interactive user access to Medium BCS occurs through three controlled paths, each enforcing R5.1 authentication:

| Access Path | Authentication | Controls |
|---|---|---|
| Local console at Control Center / substation | Unique account + password (R5.1/R5.5) | Physical access via PSP (CIP-006); session logged (CIP-007 R4) |
| Interactive Remote Access via Intermediate System | Unique account + **MFA** + encryption | CIP-005 R2; session logged; no direct ESP ingress |
| Emergency/CIP Exceptional Circumstance access | Named break-glass account, attributed | Post-use review and password rotation |

## Account Inventory & Types (R5.2)

GridPoint maintains a per-Cyber-Asset account inventory that classifies every known enabled account, so that default and generic accounts cannot hide in the estate. The inventory is reviewed each calendar quarter in step with the CIP-004 R4 access-privilege verification (Phase 03).

| Account Type | Control Applied | R5 Basis |
|---|---|---|
| Individual user account | Unique, attributable; CIP-004 authorization required | R5.1 |
| Default account (vendor-supplied) | Disabled where possible; password changed where not | R5.2 / R5.4 |
| Generic account | Eliminated where possible; otherwise attributed to named individuals | R5.2 / R5.3 |
| Shared operational account | Named authorized-individual list; quarterly review | R5.3 |
| Service / system account | Non-interactive; credential managed and rotated | R5.5 / R5.6 |
| Emergency (break-glass) account | Sealed credential; post-use review and rotation | R5.1 |

## Relationship to CIP-004 Personnel Program

CIP-007 R5 and CIP-004 (Phase 03) are complementary halves of access governance for the **142 personnel and 18 vendors** authorized to Medium BCS. CIP-004 R4 establishes that an individual has a documented need, current training, and a valid Personnel Risk Assessment before authorization; CIP-007 R5 then enforces that the resulting access is authenticated with a unique credential meeting the password parameters, and CIP-004 R5 guarantees the ability to log on is removed within 24 hours of a termination. An account without a corresponding CIP-004 authorization is a finding and is disabled.

## Technical Feasibility Exceptions

Some legacy substation relays and RTUs cannot technically enforce a strict R5 parameter (for example, a fixed short maximum password length, or an inability to lock out after failed attempts). Where a device cannot meet a strict requirement, GridPoint files a **Technical Feasibility Exception (TFE)** with ReliabilityFirst that documents the limitation, the compensating measures applied (such as enhanced physical access control at the PSP and heightened SIEM monitoring), and the plan to remediate at the next device refresh. The TFE register is reviewed alongside the CIP-010 R1 baseline library so that a device replacement automatically re-evaluates whether the exception can be retired.

## Evidence Produced

The account inventory, shared-account authorization lists, default-password change log, password-policy and lockout configuration exports, rotation evidence, and any TFEs are the standing artifacts presented against the CIP-007 RSAW at the ReliabilityFirst audit and constitute the **GAP-16** closure.

## Common Pitfalls Avoided

| Pitfall | GridPoint control |
|---|---|
| Vendor default password left in place on a relay | Default-password change at commissioning; TFE where infeasible (R5.4) |
| Shared operator account with no attribution | Named authorized-individual list per shared account (R5.3) |
| Password policy set in directory but not on device-local accounts | Per-asset capability register; TFE for incapable devices (R5.5/R5.6) |
| Brute-force attempts undetected | Lockout after 5 attempts or SIEM alert (R5.7 + CIP-007 R4) |
